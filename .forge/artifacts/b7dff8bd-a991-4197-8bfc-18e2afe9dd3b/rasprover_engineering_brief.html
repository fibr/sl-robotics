1. Platform Overview

The Waveshare RaspRover is a compact, open-source 4-wheel-drive mobile robot platform designed for learning and prototyping AI robotics — covering computer vision, autonomous navigation, SLAM, and manipulation. It is positioned as an affordable research vehicle that accepts a Raspberry Pi 4B or 5 as its "upper computer" (high-level AI/strategy) and uses an onboard ESP32 as the "lower computer" (real-time motion and sensor I/O).

It occupies a niche between hobby chassis kits (e.g., Elegoo, ELEGOO Otto) and research platforms (e.g., TurtleBot 4): it ships pre-assembled with metal chassis, encoders, IMU, camera, and a web UI, yet exposes raw serial JSON and a full ROS 2 (Humble) stack for custom development.

ℹ
Naming note Waveshare sells a family of related platforms — WAVE ROVER (bare chassis, no SBC), RaspRover (Raspberry Pi + 4WD, smaller/slower), and UGV Rover (6-wheel 4WD, faster, same software stack). The RaspRover is the entry-level complete kit. All share the same ugv_rpi software repository.

Design Goals (as stated by Waveshare)

  • Accessible AI robotics education: JupyterLab + Python tutorials, web UI out-of-the-box
  • Dual-controller separation: RPi handles vision/ML; ESP32 handles real-time motor PID + IMU
  • Expandable research base: LiDAR, robotic arm (RoArm-M2), depth camera, 4G/5G add-ons
  • ROS 2 Humble integration for advanced SLAM and autonomous navigation

3.3 ESP32 Control Protocol

All commands sent from RPi → ESP32 over UART as UTF-8 JSON lines (newline-terminated, 115200 baud):

# Drive: left/right speed [-255, 255]
{"T":1,  "L":100,  "R":100}

# Pan-tilt (gimbal)
{"T":133, "X":0.0, "Y":0.0, "SPD":0, "ACC":0}

# LED brightness (PWM, 0-255)
{"T":132, "IO4":128, "IO5":0}

# OLED text
{"T":3,  "lineNum":0, "Text":"Hello"}

# Robot type config (main=2 is RaspRover, module=0 = nothing, 2 = pan-tilt)
{"T":900, "main":2, "module":2}

# ESP32 → RPi feedback (type 1003 = emergency status)
{"T":1003, ...}

3.5 ROS 2 Integration

ROS 2 Humble (LTS) is the official supported version. The ugv_base_ros firmware makes the ESP32 behave as a ROS driver node. Documented capabilities include:

  • 2D mapping: LiDAR-based gmapping / Cartographer SLAM
  • 3D mapping: Depth camera point cloud (requires optional depth camera add-on)
  • Auto navigation: Nav2 stack integration (costmaps, path planners)
  • Gazebo simulation: URDF + Gazebo sim described in tutorial series
  • Teleop: keyboard, gamepad, web UI
ℹ
ROS 2 availability caveat The ROS 2 feature set is better documented for the UGV Rover (6-wheel) than for RaspRover specifically. Because both run the same ugv_rpi/ugv_base_ros stack, feature parity is expected, but the RaspRover-specific ROS 2 wiki section should be reviewed before assuming full Nav2/SLAM support without LiDAR add-on.

4. Practical Bring-Up Plan

4.1 Prerequisites (before first power-on)

  1. Batteries: Purchase 3× 18650 lithium cells (e.g., Samsung 25R or equivalent, ≥2500 mAh, with protection circuit). These are mandatory — the robot will not run without them.
  2. Raspberry Pi: Not included in the "Acce" (accessories-only) variant. Pi 5 4 GB is recommended for performance margin. Confirm the kit variant purchased.
  3. MicroSD (if fresh Pi): 32+ GB Class 10. Use the Waveshare pre-flashed image for fastest start, or install Debian Bookworm + ugv_rpi manually.
  4. Network: The robot auto-creates a WiFi hotspot on first boot. Have a laptop/phone to connect to it. Alternatively, pre-configure wpa_supplicant.conf to join your lab network.

4.4 Verified Control Interfaces

  • Web UI: Browser at :5000 — keyboard, gamepad, mouse
  • Python direct serial: Import base_ctrl.py, instantiate BaseController, call send_speed(left, right)
  • REST API: Flask exposes HTTP endpoints for motion commands (documented in tutorial notebooks)
  • ROS 2 topics: Once ROS driver active, standard /cmd_vel, /odom, /scan topics
  • JupyterLab: Interactive notebook control at :8888
  • Wireless gamepad: Included USB gamepad works out of the box

4.5 What to Verify First

  1. Serial link health: Send a speed command and confirm JSON echo from ESP32. This is the most critical path — all motor, servo, and sensor data flows here.
  2. Motor direction: Verify left/right motor conventions match config. On skid-steer, flipped motor wiring causes circular motion instead of straight driving.
  3. IMU axes: Confirm ICM20948 orientation (X/Y/Z) relative to robot frame matches config.yaml settings and any URDF files.
  4. Camera FoV & focus: At 160° FoV, verify there's no excessive barrel distortion in calibration use cases; the wide angle may need undistortion for precise measurements.
  5. Power budget: Monitor INA219 current with all subsystems active. N20 motors + RPi 5 + servos under load may approach or exceed the UPS module's continuous output.

5. ArmRobotics Integration Considerations

The existing sl-robotics-dev1 codebase targets a stationary Nova5 / arm101 arm with RealSense D435i and has a mature Python stack (Pinocchio IK, PyQt5 GUI, ROS2 optional). The RaspRover represents a mobile manipulation addition — a rover that can carry a camera (and potentially a small arm) to objects, rather than objects being placed in front of a fixed arm.

5.3 Suggested Integration Pattern

Rather than merging the RaspRover software wholesale into sl-robotics-dev1, the recommended approach is:

  1. Add a src/robot/rasprover_api.py driver following the same abstraction as dobot_api.py — wraps serial JSON commands as Python methods (move(speed_l, speed_r), set_pan_tilt(x, y), get_imu())
  2. Register robot_type: rasprover in robot_config.yaml and extend connect_robot() in config_loader.py
  3. Add a --view rasprover_control panel in the unified PyQt5 GUI
  4. Treat the RaspRover's built-in web UI (:5000) as a diagnostic tool, not the primary control path in our repo

7. Recommended First Forge Implementation Tasks

  • 1
    S Hardware acquisition & first-boot validation
    Purchase RaspRover PT PI5 Kit Acce + Pi 5 4 GB + 3× 18650 cells + 12.6 V charger. Boot, confirm web UI at :5000, WASD drive, camera feed. No code changes needed.
  • 2
    S Serial link test script
    Write a minimal Python script in scripts/test_rasprover_serial.py: open UART, send stop command, send forward/backward/spin, print IMU JSON feedback. Verify all motors respond correctly and feedback parsing works. This is the foundation for all subsequent integration.
  • 3
    M Add src/robot/rasprover_api.py driver
    Implement a Python class following the same abstraction pattern as dobot_api.py. Methods: connect(), move(L, R), stop(), set_pan_tilt(x, y, spd), set_led(io4, io5), get_imu(), get_battery(). Register as robot_type: rasprover in robot_config.yaml.
  • 4
    M Unified GUI: RaspRover control view
    Add --view rasprover_control to unified_gui_pyqt.py. Include: directional joypad widget (mapped to L/R speed), pan-tilt sliders, LED brightness slider, live camera feed (USB webcam via OpenCV), IMU readout. Pattern after the existing arm101 control view.
  • 5
    M IMU calibration & odometry pipeline
    Run Waveshare's IMU calibration notebook to zero-out ICM20948 biases. Then build a simple dead-reckoning odometry module in src/robot/rasprover_odom.py: integrate wheel encoder counts (via feedback JSON) + IMU to estimate pose. Verify against measured ground truth (mark floor, compare estimated vs actual travel).
  • 6
    L Camera + arm101 hand-eye calibration on mobile base
    Mount the RealSense D435i on the RaspRover (using GPIO/USB from Pi 5) alongside the existing 5MP camera. Extend src/calibration/ to handle a mobile-base extrinsic: rover→camera transform. Use existing hand-eye calibration workflow as starting point. Determine whether depth from RealSense is needed on the rover or if 5MP monocular suffices for rod detection at range.
  • 7
    L ROS 2 bring-up: cmd_vel + /scan integration
    Install D500 LiDAR on rover. Configure ugv_base_ros ROS driver on ESP32. Write a ROS 2 launch file for the rover in src/ros/rasprover_bringup.launch.py. Verify /cmd_vel drives the rover, /scan publishes LiDAR data. Run a simple SLAM demo with gmapping. This unlocks autonomous navigation for the hackathon pipeline.
